# Hiring-grade code audit — mikkonumminen.dev

**Date:** 2026-05-07
**Scope:** all source under `src/`, plus root config and `public/`. ~10,400 LOC.
**Method:** six parallel sub-agents, each scoped to a category cluster, then consolidated.
**Output:** read-only review. No edits made to the codebase.

---

## 1. Executive summary

This is a strong portfolio repo. The Astro / Three.js / GSAP stack is used with discipline that most freelance "design portfolios" never reach: explicit dispose paths for WebGL resources, GSAP context-and-revert cleanup, JS-driven reduced-motion gating in addition to CSS, sophisticated focus management on the projects drawer, comprehensive i18n with three locales, and a thoughtful Vercel security-headers policy with CSP rationale documented in the README. There is **zero unjustified `any`**, **zero `console.log`** in production paths, and the `noUncheckedIndexedAccess` discipline is honored throughout.

**Biggest risk:** every canonical URL on the live site (sitemap, OG cards, JSON-LD `Person.url`, `og:url`, hreflang alternates, `robots.txt` sitemap reference, the `portfolio.liveUrl` field in `projects.ts`) points to the Vercel preview host `mikkonumminen-dev.vercel.app` rather than the production domain. A recruiter who shares a link generates a card pointing at the preview deployment. Search engines are told the canonical is the preview. This is a one-config-line change with five-place propagation, but it must happen before the custom domain is promoted.

**Second-biggest risk:** there is no `404.astro`. Mistyped paths fall through to Vercel's default 404 — unstyled, no nav, no language handling. For a portfolio whose entry point is recruiter-typed URLs, the 404 is a real surface.

Beyond those two blockers, the report below contains **17 Major** findings (memory leaks in the projects scene, terminal teardown gaps, three CSS files missing per-file reduced-motion blocks, a 421-LOC Astro file mixing five concerns, a duplicated `escapeHtml` implementation across two unrelated subsystems, etc.) and a long tail of polish items. None of them block a recruiter visit; all of them are visible to a senior reviewer who reads the source.

**Strengths worth keeping:** `gsap.context` + `revert()` everywhere, `createScope` helper, comprehensive resource disposal in `homeScene.ts`, focus-trap + escape-to-close on the projects drawer, `bfcache`-aware page transition, `prefers-reduced-motion` honored in JS-CSS-and-design (with three gaps noted), no `client:*` directives (correct for the MPA-with-hard-navigation model), lean dependency tree.

## 3. Per-category narrative

### Project hygiene & tooling
README is excellent — the strongest recruiter-facing asset in the repo. CSP rationale and full deployment story are documented honestly. `.gitignore` covers AI tooling (`.claude/`, `.cursor/`, `.aider*`, `.windsurf/`) and editor cruft properly. `package.json` is tight: no unused deps, no duplicate-purpose libs, `sharp` and `prettier-plugin-astro` correctly in devDeps. The two real gaps are **no ESLint** and **no CI workflow** — both visible omissions on a hiring-grade repo. The `site` URL pointing at the Vercel preview is the highest-impact single fix.

### Tooling: TypeScript
`astro/tsconfigs/strict` + `noUncheckedIndexedAccess` is the right baseline; the four "stricter strict" flags (`noUnusedLocals`, `noUnusedParameters`, `noImplicitOverride`, `noFallthroughCasesInSwitch`) are absent and would catch real classes of issues. The codebase is otherwise type-disciplined: no unjustified `any`, every `as` cast carries a comment justifying it, `noUncheckedIndexedAccess` is honored (`arr[i3 + 1]!` with in-bounds proof, `tokens[0] ?? ''` coalescing, etc.).

### Architecture & cross-file consistency
File and identifier naming are uniformly good (`sphericalCurrent`, `dragMoved`, `goatInitialized`). PascalCase Astro components, camelCase utilities — no drift. Folder structure is conventional. The two structural issues are: **the `~/*` alias is defined but used 0 times** (dead config), and **`PLANET_BASE_RADIUS`, `escapeHtml`, and `Theme` are duplicated across files** (drift hazard). The single 421-LOC outlier is `ProjectsPage.astro`, which mixes five concerns.

### Three.js
The home scene's dispose path is exemplary (full geometry / material / texture / listener cleanup, DPR cap, reduced-motion gating, visibilitychange handling). The projects scene is mostly equally clean but has three real leaks: the per-frame `LineGeometry.setPositions` GPU buffer reallocation, the `_sharedPulseTexture` non-refcounted dispose, and the `Line2` instance buffer not being disposed. The fourth issue — `resize.handler()` not fired on mount — is currently masked by `createRenderer` sizing the canvas correctly, but breaks `LineMaterial.resolution` if the user resizes between page load and the deferred (≤ 2 s) WebGL boot.

### GSAP
Strong. `createScope` + `context.revert()` is used consistently. The recently-added `gsap.ticker.add(tickGoat)` is correctly removed before `scope.dispose()`. The bare `gsap.to(scale, ...)` calls in `projectsScene.ts` hover handler are covered by explicit `gsap.killTweensOf` on dispose — not a leak, but inconsistent with the context pattern.

### Terminal & page transition
Command parser is XSS-safe: every user-controllable string flows through `escapeHTML` before touching the DOM. Tab completion, history bounds (100), bash-style dedup, modifier-key bypass on link interception, and `bfcache` (`pageshow`) handling are all correct. The gaps are around **teardown** — `initTerminal` doesn't return a `dispose()`, the `measureSpan` is leaked across re-inits, and `pageTransition.ts` has a navigation-race RAF orphan on rapid double-clicks. Hard navigation hides the leaks today; any future SPA model would expose them.

### Data
Project IDs unique. Connection endpoints all reference real projects (no orphans). Timeline altitudes strictly monotone (0.08 → 0.97). External-API strings consistently title-cased. The single concrete bug is `portfolio.liveUrl` pointing at the Vercel preview host — same root cause as Blocker 1.

### Astro pages, layouts, components
Zero `client:*` directives — correct for the MPA-with-hard-navigation model. No `<img>` or `<Image />` (the site is SVG + canvas), so image-related concerns are moot. Skip-link, `<html lang>`, semantic landmarks, focus-visible all present. Sophisticated focus-trap on the projects drawer with focus-restore. The four real issues are: missing 404, duplicate `ProjectGrid` for noscript, SVG SMIL bypassing reduced-motion, and `outline: none` on the `#main` skip-link target.

### i18n
Locale parity is structural-perfect (every key in en exists in fi and sv with matching shape). One value-level bug: `cmdWhoamiCommunity` is `'community'` in sv (should be `'gemenskap'`). The hardcoded `/(fi|sv)/` regex in `localizePath` is a mild scaling smell. No HTML in translation strings, no hardcoded class names — clean separation of concerns.

### Styles
Token discipline is partial: terminal colors and fonts are tokenized via `@theme` and used with `var()` consistently, but the experience accent (`#d4ff80`) and projects accent (`#80a8ff`) are raw hex repeated across 4–5 files each. Reduced-motion coverage is partial: 6 of 9 CSS files have proper per-file blocks; **3 don't** (`nav-cards.css`, `project-detail.css`, `project-grid.css`) — covered by the global `transition-duration: 0.001ms` blanket but their initial `transform` offsets still produce one-frame jumps under reduced-motion.
